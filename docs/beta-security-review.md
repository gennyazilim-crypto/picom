# Picom Beta Security and Privacy Review

## Review scope

This review covers the `0.1.1-beta.1` Electron desktop candidate for Windows, Linux, and macOS. It is a source/configuration review plus deterministic smoke validation, not a penetration test or a claim that the remote production environment is certified.

## Result summary

- No committed runtime password, access token, Supabase service-role key, LiveKit secret, signing secret, or authorization header was detected by the repository smoke scan.
- Electron keeps `contextIsolation: true`, `nodeIntegration: false`, a sandboxed renderer, and a minimal preload bridge.
- No `dangerouslySetInnerHTML` message rendering path was found. Message bodies are rendered as React text/link parts.
- External message, OAuth, help, and status links use `externalLinkService`; unknown and unsafe protocols are rejected.
- Image uploads are limited to PNG/JPEG/WEBP/GIF, validated by MIME type, extension, and a 10 MB limit in both renderer metadata validation and the protected Edge Function boundary.
- LiveKit tokens are generated by the server-side Supabase Edge Function. `LIVEKIT_API_SECRET` is read only in that function and is not present in renderer configuration.
- Diagnostics and feedback payloads pass through central redaction and remain explicit user actions.
- Public registration requires acceptance of the beta Terms of Service and Privacy Policy placeholders.

No source-level critical blocker was found. Live environment verification remains mandatory before broader connected testing.

## Secret and environment boundaries

### Renderer-safe

- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`
- Public OAuth availability flags and callback URL
- Public LiveKit WebSocket URL and availability flag
- Public release/version/status configuration

The Supabase anon key is not a privileged secret and is safe only when RLS is enabled and verified.

### Server/CI secret store only

- Supabase service-role key
- Supabase access token and database password
- LiveKit API key/secret
- Signing certificates/passwords and notarization credentials
- Private OAuth provider secrets
- Database, Redis, email, or monitoring credentials

These values must never be prefixed with `VITE_`, embedded in the renderer, copied into diagnostics, or committed.

## Authorization and RLS review

Static migration and pgTAP-shape smoke coverage confirms policies/tests exist for:

- Anonymous users reading only public community/channel/message data allowed by public-read flags.
- Visitors being unable to send messages.
- Visitors being unable to view private channels or their messages/attachments.
- Members sending only in visible text channels.
- Message authors editing their own messages.
- Owner/admin moderation boundaries represented in policy tests.
- Storage attachment reads following visible-message/channel access.

`npm run supabase:rls:smoke` passed structurally. Supabase CLI was unavailable, so real pgTAP execution and deployed-project policy behavior remain unverified.

## Content and link safety

- Message content is rendered through React; raw HTML injection is not used.
- Detected URLs are normalized and opened through `externalLinkService`.
- Only `http` and `https` are allowed for external links; desktop deep links use their separate validated service.
- Blocked links produce a user-facing error instead of invoking native shell behavior directly.

## Upload and attachment safety

- Allowed MIME types: `image/png`, `image/jpeg`, `image/webp`, `image/gif`.
- Allowed extensions: `.png`, `.jpg`, `.jpeg`, `.webp`, `.gif`.
- Maximum size: 10 MB.
- File names are sanitized at the Edge Function boundary.
- Storage RLS ties reads to visible attached messages and prevents private-channel visitor access.

Client validation is UX only. Edge Function validation, storage policies, and RLS are the security boundary. Malware scanning remains a documented production placeholder and is not claimed as active.

## Voice and screen share

- Token issuance is server-side and requires the Edge Function secret store.
- Renderer receives only a short-lived room token and public URL needed to join.
- Electron screen capture is exposed through validated preload IPC, not raw Electron APIs in React.
- OS microphone/screen-recording permission denial and recovery must be verified on each target OS.

## Diagnostics and privacy

- Redaction covers password/token/cookie/key/authorization-like fields.
- Recent logs are bounded and included only when the user opts in.
- Feedback export is local/manual; no automatic support upload exists.
- Private message content is not intentionally included in diagnostics.
- Operators must inspect exported reports before forwarding them.

## Public registration and legal notice

`RegisterScreen` requires the user to accept the beta Terms of Service and Privacy Policy before account creation. The documents remain beta placeholders and require legal approval before a public stable release.

## External environment checks required

Before connected beta expansion:

1. Install Supabase CLI and run `npm run supabase:rls:test` against an isolated local/staging environment.
2. Execute the full owner/admin/moderator/member/visitor/private-channel matrix using distinct accounts.
3. Confirm the deployed Storage bucket is private and object policies match message visibility.
4. Confirm Realtime cannot subscribe a visitor to private-channel rows.
5. Verify Edge Functions require valid JWTs and expose no secrets in errors/logs.
6. Rotate any credential ever used in an untrusted local environment.
7. Verify LiveKit tokens are short-lived, room-scoped, identity-bound, and cannot grant admin permissions.
8. Test microphone and screen-capture denial/recovery on native Windows, Linux, and macOS hosts.
9. Obtain legal approval for Terms/Privacy text before public registration.

## Release blockers

- Any private community/channel/message/attachment visibility leak.
- Visitor message insert/reaction/upload success.
- Service-role or LiveKit secret exposure in renderer, logs, artifacts, or reports.
- Authentication/session bypass or cross-user data access.
- Unsafe external protocol execution.
- Unredacted credentials in exported diagnostics.

The connected beta must remain gated until the live RLS/Storage/Realtime checks above pass.
