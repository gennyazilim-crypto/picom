# Developer Portal v1

## Status

Picom Developer Portal v1 is a feature-flagged, restricted desktop view for safe bot/webhook metadata and architecture documentation. It does not expose a public platform, API keys, raw tokens, application publishing, or executable integrations.

## Access

The portal entry is shown only when both conditions hold:

1. `enableDeveloperPortal` is enabled by approved environment/remote configuration.
2. The current user can manage bots (`manageCommunity`) or webhooks (`manageChannels`) in the active community.

The feature flag defaults to off. Production runtime overrides are rejected by `featureFlagService`. Deep rendering is guarded again inside `DeveloperPortalView`, so a hidden entry point is not the only availability check.

Frontend checks are UX boundaries. Bot/webhook mutations continue to require backend/RLS authorization in their own services.

## Sections

### My Bots

- Lists bot display name, assigned role, credential configured/revoked state, and non-secret token prefix.
- Never displays raw bot tokens or token hashes.
- Directs creation, role assignment, token creation, and revocation to Community Settings > Bots.
- No executable runtime, dynamic loading, or marketplace is included.

### Webhooks

- Lists safe webhook name and active/revoked status for the active community.
- Uses `webhookService.list()`, whose Supabase select excludes `token_hash`.
- Never displays one-time URLs, raw tokens, token hashes, request payloads, or private content.
- Directs create/revoke actions to Community Settings > Webhooks.

### Applications placeholder

- Clearly marks application registration, OAuth clients, public publishing, review, and marketplace listing as unavailable.
- Does not create database applications or issue credentials.

### API Docs placeholder

- Summarizes bot and webhook foundation status.
- Does not expose a production API base URL, service-role configuration, provider key, or secret.
- Points future work toward versioned, scoped, rate-limited, audited APIs.

## Security rules

- No raw API keys, bot tokens, webhook tokens, token hashes, authorization headers, or Supabase/LiveKit secrets.
- No public publishing or unreviewed application discovery.
- No arbitrary code execution, `eval`, dynamic imports from integrations, shell, filesystem, process, native module, or Electron IPC access.
- No private message/attachment/event data unless the integration's backend role/RLS explicitly authorizes it.
- No renderer direct access to backend-only bot credential or webhook delivery RPCs.
- No normal-member entry point.

## Desktop behavior

- Opens as a desktop modal using Picom's existing Settings layout tokens.
- Uses Coolicons through `AppIcon`.
- Closes on Escape, close button, or backdrop click.
- Does not alter ServerRail, community/chat layout, Mention Feed, Profile, titlebar, or window controls.
- No mobile layout or public web route is added.

## Current limitations

- Scope is the active community rather than a global cross-community developer account view.
- Bot metadata uses the existing local foundation; production Bot API provisioning remains unavailable.
- Webhook metadata depends on current manager/RLS access.
- Applications and API Docs are explicit placeholders.
- There is no public publishing, app review, OAuth client, SDK, or production API key issuance.

## Future gates

- Developer account and organization ownership model.
- Protected application schema and RLS.
- Scoped one-time credential issuance through trusted backend functions.
- API versioning, quotas, idempotency, audit, abuse controls, and observability.
- Review/signing and publishing policy.
- Legal/privacy and support readiness.
- Security review before enabling the feature flag in stable builds.
