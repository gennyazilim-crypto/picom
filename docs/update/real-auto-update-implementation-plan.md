# Real Auto-Update Implementation Plan

## Status

Proposed, disabled by default. Picom does not enable production auto-update as part of this plan.

## Goals

- Deliver signed Picom desktop updates through an auditable Electron update path.
- Keep development, beta, and stable users on explicitly selected release channels.
- Verify update metadata and artifacts before installation.
- Recover safely from download, verification, installation, or post-update failures.
- Keep credentials, signing keys, and publishing tokens outside the repository and renderer process.

## Non-goals

- Enabling automatic background installation now.
- Adding an updater endpoint or publishing production artifacts.
- Committing code-signing certificates, notarization credentials, or hosting tokens.
- Claiming uniform self-update behavior for every Linux package format.

## Proposed updater

Use `electron-updater` with `electron-builder` when the release pipeline, signing, and artifact hosting prerequisites are ready. It supports the existing Electron packaging direction and established update metadata formats.

Before adoption, pin a reviewed compatible version, audit transitive dependencies, and prototype in an isolated beta channel. The renderer must continue to call `updateService`; only the Electron main process may own updater APIs. Preload should expose a narrow, validated status/action bridge and never raw Electron updater objects.

## Release channels

| Channel | Audience | Update source | Policy |
| --- | --- | --- | --- |
| `dev` | Local developers and internal test builds | Local/manual artifacts or disabled updater | Never points at production; unsigned local builds remain manual. |
| `beta` | Opt-in testers | Dedicated beta manifest and signed prereleases | Staged first, pauseable, and allowed to contain known non-blocking issues. |
| `stable` | General release users | Stable manifest containing promoted artifacts only | Requires full signing, platform certification, rollback readiness, and Go/No-Go approval. |

Channel selection must be stored as non-sensitive settings and validated in the main process. Remote config may pause a channel or lower rollout eligibility, but it must not bypass artifact signature verification.

## Update manifest

The release pipeline should generate platform-specific metadata supported by the chosen updater. Each published entry should identify:

- Picom semantic version.
- Release channel.
- Publication timestamp.
- Artifact filename and supported architecture.
- SHA-512 value required by `electron-updater` metadata.
- SHA-256 checksum for independent user/release verification.
- Artifact size.
- Release notes URL or sanitized summary.
- Minimum supported client version when applicable.
- Rollout percentage placeholder when staged rollout is implemented.

The manifest must be generated by CI from completed artifacts, uploaded atomically after artifacts, and served over HTTPS. It must contain no credentials or private infrastructure details.

## Artifact hosting

Preferred options are a private-to-public promotion flow in an object store/CDN or a supported release provider. The final choice must provide:

- HTTPS and immutable versioned artifact paths.
- Correct MIME types and byte-range support.
- Atomic manifest publication.
- Access logs without query-string secrets.
- Retention of at least the current and previous approved version.
- A documented method to pause manifest publication immediately.
- Separate paths or buckets for `beta` and `stable`.

Publishing credentials live in CI secret storage and are scoped to release upload only. Desktop clients receive only public download locations.

## Windows prerequisites

- Acquire an organization-controlled code-signing certificate appropriate for Windows distribution.
- Store certificate material in a managed signing service or protected CI secret store; never commit a PFX or password.
- Sign the executable, installer, and relevant update artifacts with timestamping.
- Verify publisher identity, signature validity, installation, update, and rollback behavior on supported Windows versions.
- Confirm SmartScreen reputation and first-install behavior during beta certification.

Production Windows auto-update remains blocked until repeatable signing and verification are working in CI.

## macOS prerequisites

- Use an Apple Developer ID Application identity controlled by the organization.
- Enable hardened runtime and declare only required entitlements.
- Sign all nested binaries correctly.
- Notarize each distributable and staple the notarization ticket where applicable.
- Verify Gatekeeper behavior on a clean supported macOS installation.
- Keep App Store Connect API keys, certificate material, and keychain credentials in managed CI secrets.

Production macOS auto-update remains blocked until signing, notarization, stapling, and update installation are certified for shipped architectures.

## Linux limitations

Linux update behavior depends on package format and distribution policy:

- AppImage may support updater-assisted replacement if metadata and hosting meet updater requirements.
- `.deb` and `.rpm` installations are normally updated by their package manager or repository; in-app replacement should not bypass system ownership.
- Sandbox/package formats such as Flatpak or Snap must use their own update channels if adopted.
- Picom should show update availability and direct users to the correct trusted mechanism when self-update is unsupported.

The UI must not claim that an update was installed when only a download page was opened.

## Runtime architecture

1. Renderer requests a check through `updateService`.
2. Preload forwards only a whitelisted command to the main process.
3. Main process selects the configured channel and checks the trusted manifest.
4. Main process emits normalized, non-sensitive update states.
5. Renderer presents status and user choices without handling artifact paths or credentials.
6. Installation occurs only after integrity/signature validation and an explicit approved policy.

Recommended normalized states:

- `idle`
- `checking`
- `update_available`
- `up_to_date`
- `downloading`
- `downloaded`
- `install_pending`
- `download_failed`
- `verification_failed`
- `install_failed`
- `paused_by_rollout`
- `unsupported_platform`
- `rollback_available_placeholder`

Errors shown to users must be concise. Detailed updater errors belong only in redacted diagnostics.

## Rollout, pause, and rollback

- Publish to `dev`, then a small internal `beta` ring, then broader beta, then stable.
- Keep the stable manifest unchanged until platform smoke tests pass.
- Pause by removing rollout eligibility or restoring the last known-good manifest pointer; do not overwrite immutable artifacts.
- Use feature flags and emergency kill switches to disable risky application features independently of the updater.
- Retain the prior signed installer and compatibility metadata.
- Do not automatically downgrade user data or database schemas.
- Before rollback, verify the previous client remains compatible with the current backend and local data schema.
- If a bad update prevents normal startup, direct users to safe mode, log export, and a signed previous installer according to the update failure recovery runbook.

## Failure handling

| Failure | Required behavior |
| --- | --- |
| Manifest unavailable | Keep current app working; report a non-blocking check failure. |
| No eligible rollout | Remain on current version and show no misleading error. |
| Download interrupted | Preserve app usability; allow a safe retry later. |
| Checksum/signature mismatch | Delete or quarantine the artifact, block installation, and log a redacted security event. |
| Insufficient disk space | Stop before installation and provide actionable guidance. |
| Install failure | Keep or restore the current usable installation where the platform installer supports it. |
| App fails after update | Offer safe mode/recovery and pause rollout while the incident is assessed. |
| Unsupported Linux package | Show update information and the appropriate trusted package/repository path. |

## Security boundaries

- Updater operations run in the Electron main process, never React components.
- `contextIsolation` remains enabled and `nodeIntegration` remains disabled.
- IPC channels and payloads are whitelisted and validated.
- HTTPS is required, but transport security does not replace artifact signature/integrity checks.
- Release manifests, build metadata, and logs contain no secrets.
- CI signing identities use least privilege, protected branches/environments, and auditable approvals.
- A compromised remote configuration service cannot disable signature validation or grant native capabilities.

## Implementation phases

1. Select hosting provider and document retention/pause behavior.
2. Establish Windows signing and macOS signing/notarization in protected CI.
3. Add pinned updater dependency in an isolated change and implement main/preload adapters behind `updateService`.
4. Generate and validate manifests in CI without publishing to stable.
5. Certify download/install/restart/failure behavior on Windows, Linux package formats, and macOS.
6. Enable opt-in internal `dev` checks.
7. Enable beta channel with manual installation approval and monitored rollout.
8. Complete security/release review before any stable auto-update enablement.

## Production enablement gates

- Signed and notarized artifacts pass clean-machine installation tests.
- Manifest and artifact integrity checks have negative-path tests.
- Update IPC security review passes.
- Rollout pause and previous-version recovery are rehearsed.
- Client/server and local-data backward compatibility are documented.
- Monitoring distinguishes check, download, verification, install, and post-update startup failures.
- Support documentation and incident ownership are assigned.
- Final Go/No-Go explicitly approves stable auto-update.
