# Two-factor authentication production decision

Decision: **defer production 2FA to post-MVP security work**. Picom's current Settings controls are explicitly `Coming soon`/local placeholders and provide no authentication protection. They must not be presented as enabled MFA, used for authorization, required by policy, or included in security/compliance claims. No TOTP secret, QR payload, recovery code or factor identifier is generated by the current app.

## Why it is deferred

A safe rollout needs a complete Supabase Auth MFA challenge lifecycle, not an enrollment toggle. Picom does not yet have hosted AAL1/AAL2 tests, cross-device challenge UX, factor lifecycle, recovery-code backend, support identity-proofing policy, lockout recovery, rate/abuse monitoring, independent security review or Windows/Linux/macOS certification. Half-enabling enrollment would create account lockout and false-security risk.

## Required implementation contract

- Use Supabase Auth's reviewed TOTP/MFA APIs; never implement custom OTP cryptography.
- Require a recent password re-authenticated AAL1 session before enroll, unenroll, factor replacement or recovery-code rotation.
- Show the provider enrollment QR/secret only during the active enrollment step, never in logs, diagnostics, analytics, clipboard history, screenshots, localStorage or application tables.
- Enrollment is not active until a valid TOTP challenge is verified and the server reports AAL2.
- Login/session restore must detect an MFA challenge and block protected app data until AAL2 succeeds. Rate-limit attempts and use neutral errors.
- Factor IDs are safe metadata only when scoped to the owner; raw TOTP seeds never reach Picom persistence.
- Recovery codes are generated with a reviewed provider/backend process, displayed once, stored only as strong server-side hashes, individually consumed/revoked, rate-limited and rotated after suspected compromise.
- Disabling/replacing factors or regenerating recovery codes revokes other sessions where policy requires and appends content-free account security events.
- Support cannot view seeds/codes or bypass MFA casually. Recovery requires a documented, privacy-reviewed identity-proofing and escalation process with anti-social-engineering controls.

## Rollout gates

Production remains no-go until hosted staging verifies enrollment/challenge/unenrollment, wrong/replayed/expired codes, rate limits, device clock skew, offline behavior, session refresh/revocation, password reset, account deletion, lost-device recovery and cross-client synchronization. Product/security/privacy/legal/support must approve user copy, recovery policy and risk ownership. Independent review and rollback/kill-switch plans are required before `REQUIRE_MFA` or equivalent enforcement exists.

Until then, Picom keeps MFA unavailable, stores no sensitive MFA material and directs users to strong unique passwords plus existing session controls.
